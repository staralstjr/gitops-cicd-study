# 개념 04 — 배포 구조 파악: 앱 레포 vs 인프라 레포, 그리고 "환경"

> 기존 배포 문서를 분석해 **CI(빌드)와 CD(배포)의 경계**, 그리고
> "환경(environment)"이라는 단어가 실제로 무엇을 가리키는지 정리한다.
> 이름(namespace/registry)만 보고 환경을 넘겨짚지 않는 것이 핵심 교훈.

선행 개념: [concept-02-gitops-argocd.md](concept-02-gitops-argocd.md)

---

## 1. 앱 레포와 인프라 레포는 분리되어 있다

컨테이너 기반 배포에서는 **애플리케이션 코드**와 **인프라 선언**을 서로 다른 저장소로 나눈다.

```
┌──────────────────────────────────────────────────────────┐
│ 앱 레포 (web-frontend / web-api)                           │
│   소스코드 → 이미지 빌드 → 레지스트리 push          = CI    │
│   "실행 가능한 결과물(이미지)을 만든다"                       │
│   ※ 이미지를 만들 뿐, 스스로 배포하지 않는다                  │
└──────────────────────────────────────────────────────────┘
                     │  이미지가 레지스트리에 쌓임
                     ▼
┌──────────────────────────────────────────────────────────┐
│ 인프라 레포 (k8s-infra)                                    │
│   "어떤 이미지를, 몇 개 파드로, 어떤 설정으로 띄울지" 선언   = CD│
│   Argo CD가 이 레포의 특정 브랜치를 감시 → 클러스터에 반영    │
│   "여기가 진짜 '배포' 트리거"                                │
└──────────────────────────────────────────────────────────┘
```

**핵심:** 앱 레포의 브랜치(dev 등)는 **"언제 이미지를 빌드할지"** 만 정한다.
실제 **"배포"** 는 인프라 레포의 매니페스트가 바뀌고 Argo CD가 이를 감지할 때 일어난다.
→ 두 레포의 트리거는 **분리**되어 있다.

### 분석 대상 배포 문서에서 확인된 사실

| 항목 | 확인 내용 |
|------|-----------|
| CD 트리거 | Argo CD Application의 `targetRevision: main` → **인프라 레포의 main 브랜치**를 감시 |
| 이미지 태그 | `:latest` 사용 → 매니페스트 텍스트가 안 변함 → **수동 restart 필요** (개념 01의 문제) |
| 클러스터 인증 | `imagePullSecrets: <regcred>` → 클러스터가 레지스트리에서 이미지를 pull할 때 사용 |

## 2. "환경(environment)"이란 무엇인가

같은 애플리케이션을 **목적이 다른 여러 벌**로 나누어 운영하는 것.

| 환경 | 용도 | 사용자 | 장애 영향 |
|------|------|--------|-----------|
| **개발(dev)** | 개발 중인 코드를 배포해 테스트 | 개발자, 내부 | 낮음 |
| **스테이징(staging)** | 프로덕션과 동일 구성으로 최종 검증 | QA, 내부 | 낮음 |
| **프로덕션(prod)** | 실제 사용자가 쓰는 서비스 | 실제 고객 | **매우 높음** |

각 환경은 **분리된 서버/클러스터/DB**를 갖는다. dev에서 무엇을 망가뜨려도 실제 사용자에게는 영향이 없다.
이것이 환경을 나누는 이유 — **안전한 실험 공간**과 **실제 서비스**의 분리.

---

## 3. "빌드"와 "배포"는 다른 단계다

가장 흔한 혼동 지점.

```
[빌드(build)]  소스코드 → 이미지를 만들어 레지스트리(창고)에 올림
               = 제품을 만들어 창고에 넣음. 아직 아무도 사용하지 않음.

[배포(deploy)] 창고의 이미지를 실제 환경의 서버에서 실행시킴
               = 창고의 제품을 매장에 진열해 사용자가 쓰게 함.
```

이 프로젝트의 Phase 1(CI)은 **빌드까지**만 담당한다.
"그 이미지가 어느 환경에서 실행(배포)되는가"는 인프라 레포 쪽의 별개 문제다.

---

## 4. 브랜치 이름에는 의미가 없다 — 의미는 규칙으로 부여된다

**Git 브랜치 자체는 "이건 prod, 이건 dev" 같은 의미를 갖지 않는다.**
`main`, `dev`, `prod`는 전부 그냥 이름표다. 어느 브랜치가 어느 환경으로 나갈지는
**팀이 자동화(워크플로)로 규칙을 정해서** 생긴다.

```
git 관점:    dev, main = 커밋을 가리키는 포인터. 본질적 차이 없음.

의미 부여:   "dev push → dev환경 배포"   ← 워크플로에 우리가 명시
            "main push → prod환경 배포"  ← 워크플로에 우리가 명시
```

따라서 "prod 브랜치"가 저절로 존재하는 게 아니라, **우리가 지정**해야 한다.
단, 기존에 팀이 이미 정해둔 규칙이 있을 수 있으므로 **새로 만들기 전에 기존 규칙을 확인**해야 한다.

> 흔한 prod 트리거 방식:
> - 특정 브랜치 머지 (`main` → prod)
> - **버전 태그** (`git tag v1.2.0` → prod). 실수 배포를 막기 위해 의도적으로 태그를 찍어야만 나가게 하는 방식.

## 5. 이름으로 환경을 넘겨짚지 말 것 (핵심 교훈)

인프라의 이름은 대부분 **관례**일 뿐, 환경 정보를 담지 않는다. 실제로 겪은 사례:

### 사례 1 — 네임스페이스 접두사

배포 문서에 다음 네임스페이스들이 등장한다.

```
app-newsboy            ← 애플리케이션 네임스페이스
addon-argocd           ← 플랫폼 부가기능 (Argo CD)
addon-sealed-secrets   ← 플랫폼 부가기능 (Sealed Secrets)
```

- **`app-*`** = 실제 서비스 애플리케이션
- **`addon-*`** = 클러스터를 운영하는 도구(부가기능)

즉 `app-newsboy`의 `app`은 **"부가기능이 아니라 애플리케이션"** 이라는 분류일 뿐,
**환경(dev/prod)도, 웹/모바일 구분도 아니다.** 네임스페이스 이름은 환경 정보를 담지 않는다.

### 사례 2 — 레지스트리는 프로젝트 단위

컨테이너 레지스트리가 프로젝트별로 하나씩 존재한다(예: `newsboy`, `curvedroad`, ...).
newsboy는 **레지스트리 하나**만 사용한다.

- → 환경(dev/prod)을 **레지스트리 단위로 나누지 않았다.**
- 환경이 여러 개라면 **태그**(`:dev-*`, `:prod-*`)로 구분하거나, 아니면 **환경이 하나**일 수 있다.

**결론:** 레지스트리·네임스페이스 **이름만으로는 환경 개수를 알 수 없다.** 반드시 담당자에게 확인한다.

## 6. Public Endpoint vs Private Endpoint

레지스트리는 보통 두 개의 접근 경로를 제공한다.

| 엔드포인트 | 접근 범위 | 사용 주체 |
|-----------|-----------|-----------|
| **Public Endpoint** | 인터넷에서 접근 가능 | 외부 CI(GitHub Actions)가 push할 때 |
| **Private Endpoint** | 클라우드 내부 사설망(VPC)에서만 | 클러스터가 이미지를 pull할 때(더 빠르고 안전) |

> VPC의 **public 서브넷(인터넷 대면) vs private 서브넷(내부 전용)** 과 동일한 사고방식이다.
> "밖에서 들어오는 문"과 "내부에서만 쓰는 문"을 분리한다.

---

## 7. CI ↔ CD를 잇는 다리 (이 프로젝트의 목표)

```
Phase 1 (CI):  앱 레포 dev 머지 → SHA 태그로 이미지 빌드/push
                          │
              (다리) 인프라 레포 매니페스트의 이미지 태그를 새 SHA로 자동 갱신
                          ▼
Phase 2 (CD):  인프라 레포 main 커밋 → Argo CD 감지 → 자동 배포
```

- 현재는 `:latest`라 매니페스트가 변하지 않음 → Argo CD가 감지 못 함 → **수동 restart**.
- **SHA 태그 + 인프라 레포 자동 갱신**으로 바꾸면 매니페스트가 변함 → **자동 배포**, 수동 restart 제거.

---

## 8. 요약

1. 앱 레포(CI, 빌드)와 인프라 레포(CD, 배포)는 **분리**되어 있고 트리거도 다르다.
2. "환경"은 목적이 다른 앱의 여러 벌(dev/staging/prod)이며, 각기 분리된 인프라를 갖는다.
3. **빌드 ≠ 배포.** Phase 1은 빌드까지만 담당한다.
4. 브랜치 이름에는 의미가 없다. **규칙(워크플로)으로 의미를 부여**하며, 기존 규칙을 먼저 확인한다.
5. 네임스페이스·레지스트리 **이름으로 환경을 넘겨짚지 않는다.** (`app-`는 분류 접두사일 뿐)
6. 레지스트리의 **public/private 엔드포인트**는 VPC의 public/private 서브넷과 같은 개념.
