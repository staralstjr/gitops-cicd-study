# 개념 01 — 컨테이너 · 이미지 · 레이어 · 레지스트리 · 태그

> GitOps CI/CD 파이프라인을 이해하기 위한 첫 번째 토대.
> "왜 컨테이너인가"부터 시작해, 마지막에 **이 프로젝트의 핵심 문제(`latest` 태그)** 로 이어진다.

---

## 1. 컨테이너는 왜 생겼나 — "제 컴퓨터에선 되는데요"

프로그램은 혼자 실행되지 않는다. 항상 **주변 환경**에 의존한다.

- OS 종류와 버전
- 런타임 버전 (Python 3.10 vs 3.8, Node 20 vs 18)
- 설치된 라이브러리와 그 버전
- 환경 변수, 설정 파일

그래서 "내 노트북에선 되는데 서버에선 안 되는" 문제가 생긴다.
원인은 코드가 아니라 **환경 차이**다.

**컨테이너의 발상**

> 프로그램만 옮기지 말고, **프로그램 + 그것이 필요로 하는 환경 전체**를
> 하나의 상자에 포장해서 통째로 옮긴다.

이러면 개발 환경에서 만든 그 상자가 그대로 서버에서 실행되므로, 환경 차이가 사라진다.

---

## 2. 가상머신(VM)과 컨테이너의 결정적 차이

"환경을 통째로 포장한다"는 아이디어 자체는 VM도 하던 일이다. 그러나 **방식**이 다르다.

```
[ 가상머신(VM) ]                    [ 컨테이너 ]
┌─────────────────┐               ┌─────────────────┐
│  앱   │   앱     │               │  앱   │   앱     │
│  라이브러리 (GB)  │               │  라이브러리 (MB) │
│  게스트 OS 전체   │ ← 커널 포함    │   (게스트 OS 없음)│
├─────────────────┤               ├─────────────────┤
│   하이퍼바이저    │               │  컨테이너 런타임  │
├─────────────────┤               ├─────────────────┤
│    호스트 OS     │               │    호스트 OS     │ ← 커널을 공유
└─────────────────┘               └─────────────────┘
```

**본질적 차이**

- **VM**: 게스트 OS를 **통째로 하나 더** 실행한다(커널 포함). 완전히 독립된 컴퓨터를 하나 더 만드는 셈.
- **컨테이너**: 호스트의 **커널을 공유**하고, **프로세스와 파일시스템만 격리**한다.
  (리눅스의 namespace / cgroup 기능을 이용)

| 항목 | VM | 컨테이너 |
|------|----|----------|
| 크기 | 수 GB | 수십~수백 MB |
| 시작 시간 | 수십 초 ~ 수 분 | **1초 이내** |
| 한 호스트에 실행 가능 수 | 수 개 | **수십~수백 개** |
| 격리 수준 | 강함(커널까지 분리) | 중간(커널 공유) |
| 주 용도 | 이기종 OS, 강한 격리 | 애플리케이션 배포, 높은 밀도 |

> **왜 중요한가:** 컨테이너의 가벼움(밀도)이 마이크로서비스와 Kubernetes를 가능하게 했다.
> 서비스 수십 개를 VM으로 띄우면 메모리가 몇 배로 들지만, 컨테이너는 같은 하드웨어에 훨씬 많이 올릴 수 있다.

### 커널 공유가 낳는 실제 제약: CPU 아키텍처

컨테이너는 호스트 커널을 공유하므로, **이미지 안의 바이너리가 실행될 CPU 아키텍처와 맞아야 한다.**

- 개발 노트북: Apple Silicon = **ARM64**
- 운영 서버(K8s 노드): 대부분 **x86_64 (amd64)**

아키텍처가 다르면 실행되지 않으므로, 빌드 시 타깃을 명시한다.

```bash
docker build --platform linux/amd64 -t <image>:<tag> .
```

VM이었다면 게스트 OS가 따로 있어 덜 민감했을 문제지만, **커널을 공유하는 컨테이너에서는 필수 고려사항**이다.

## 3. 이미지 vs 컨테이너

가장 흔하게 헷갈리는 지점. 둘은 완전히 다른 것이다.

| | **이미지 (Image)** | **컨테이너 (Container)** |
|--|--------------------|--------------------------|
| 정체 | 읽기 전용 **템플릿 / 스냅샷** | 이미지로 **실행 중인 인스턴스** |
| 상태 | 정적인 파일 덩어리 | 살아있는 프로세스 |
| 개수 | 1개 | 같은 이미지로 **N개 실행 가능** |

**익숙한 개념과의 대응**

| 개념 | 템플릿(정적) | 인스턴스(동적) |
|------|--------------|----------------|
| 객체지향 | 클래스 | 객체 |
| AWS | **Launch Template** | **EC2 인스턴스** |
| 컨테이너 | **이미지** | **컨테이너** |

> AWS Auto Scaling Group이 Launch Template 하나로 서버 여러 대를 찍어냈던 것과
> Kubernetes가 이미지 하나로 파드 여러 개를 띄우는 것은 **완전히 같은 구조**다.

## 4. 이미지는 "레이어(layer)"로 쌓인다

Dockerfile의 **명령어 한 줄이 대체로 레이어 하나**를 만든다.

```dockerfile
FROM python:3.10          # 1층: 베이스 이미지 (OS + 런타임)
COPY requirements.txt .   # 2층: 의존성 목록 복사
RUN pip install -r requirements.txt   # 3층: 라이브러리 설치
COPY . .                  # 4층: 애플리케이션 소스 복사
```

**레이어의 두 가지 핵심 성질**

1. **캐시된다**
   변경되지 않은 레이어는 재사용된다. 소스코드만 수정하면 4층만 다시 만들고 1~3층은 캐시를 쓴다.
   → 재빌드가 극적으로 빨라진다.

2. **공유된다**
   여러 이미지가 동일한 베이스 레이어를 공유하므로 저장 공간과 전송량이 절약된다.

**설계에 주는 시사점 — 자주 바뀌는 것을 뒤에 둔다**

의존성 설치(`requirements.txt`, `package.json`)를 소스 복사보다 **앞에** 두면,
소스만 바뀌었을 때 의존성 설치 레이어를 캐시로 재사용할 수 있다.
순서를 반대로 하면 코드 한 줄만 고쳐도 매번 전체 의존성을 새로 설치하게 된다.

> **CI에 직결되는 개념이다.** GitHub Actions는 매번 깨끗한 러너에서 실행되므로,
> 레이어 캐시를 어떻게 살릴지가 빌드 시간을 좌우한다.

## 5. 레지스트리 — "이미지판 Git 원격 저장소"

빌드한 이미지는 서버가 가져갈 수 있는 곳에 올려야 한다. 그 저장소가 **레지스트리**다.

```
코드   :  로컬 → git push   → GitHub    → git pull   → 동료
이미지 :  로컬 → docker push → 레지스트리 → docker pull → 서버(K8s)
```

명령 이름이 `push` / `pull`로 같은 이유가 여기 있다. **구조가 같기 때문**이다.

| 종류 | 예 | 접근 |
|------|-----|------|
| 공개 레지스트리 | Docker Hub | 누구나 pull 가능 |
| 비공개 레지스트리 | 클라우드 제공 레지스트리(NCR/ECR/GCR 등) | **인증 필요** |

**비공개 레지스트리라서 생기는 두 가지 인증 지점**

1. **개발자/CI → 레지스트리**: `docker login <registry>` (Access Key / Secret Key)
2. **Kubernetes → 레지스트리**: 클러스터도 이미지를 받으려면 로그인이 필요하다.
   그래서 Deployment에 `imagePullSecrets`(레지스트리 자격증명 Secret)를 지정한다.

> 흔한 오해: "이미지를 push했는데 파드가 못 뜬다" → 대부분 **클러스터 쪽 인증(`imagePullSecrets`) 누락**이 원인이다.

### 이미지 전체 이름의 구조

```
<registry>.kr.ncr.ntruss.com / <image-name> : <tag>
└──── 레지스트리 호스트 ────┘  └─ 이미지명 ─┘  └태그┘
```

로컬에서 만든 이미지는 이름이 `<image-name>:<tag>`뿐이므로,
push 전에 **레지스트리 호스트를 포함한 이름으로 태그를 다시 붙여야 한다**(`docker tag`).
이것이 수동 절차에 `build → login → tag → push` 4단계가 있는 이유다.

## 6. 태그(tag)와 `latest` 문제 — 이 프로젝트의 핵심

### 태그는 "이름표"일 뿐이다

태그는 이미지에 붙이는 **가변적인 라벨**이다. 버전 표시에 관례적으로 사용한다.

🔴 **`latest`는 특별하지 않다.**
자동으로 최신을 가리키는 기능이 아니라, **단지 "latest"라는 문자열 관례**일 뿐이다.
그리고 **덮어쓸 수 있다(mutable)**.

```
어제 push한 :latest   ──(덮어씀)──▶   오늘 push한 :latest
```

같은 이름표인데 **내용물이 달라진다.**

### 이것이 GitOps에서 문제가 되는 이유

GitOps 엔진(Argo CD)은 **Git에 선언된 내용과 클러스터 상태를 비교**해서 동작한다.

```yaml
# Deployment 매니페스트 — 어제도 오늘도 동일
image: <registry>.kr.ncr.ntruss.com/<image-name>:latest
```

- 새 이미지를 push해도 **매니페스트 텍스트는 변하지 않는다**
- → Argo CD가 보기엔 **"변경 없음"**
- → **아무 일도 일어나지 않는다**
- → 그래서 사람이 **수동으로 restart** 해야 한다

추가 문제:
- 현재 운영 중인 파드가 **어느 커밋의 코드인지 추적 불가**
- 문제 발생 시 **롤백할 대상 버전을 특정할 수 없음**

### 해결 — 불변(immutable) 태그

이미지 태그에 **커밋 SHA**처럼 매번 달라지는 고유값을 사용한다.

```yaml
image: <registry>.kr.ncr.ntruss.com/<image-name>:1f3b8ed   # 이전
image: <registry>.kr.ncr.ntruss.com/<image-name>:217988d   # 신규 배포
```

| 효과 | 설명 |
|------|------|
| **자동 배포 가능** | 매니페스트 텍스트가 바뀌므로 Argo CD가 변경을 감지해 자동 동기화 |
| **추적성** | 운영 중인 버전이 어느 커밋인지 즉시 확인 가능 |
| **롤백 용이** | 이전 태그로 되돌리기만 하면 됨 |
| **캐시 안전** | 같은 태그가 다른 내용을 가리키는 혼란이 사라짐 |

> **이 전환이 자동화 설계의 뼈대다.**
> `latest` → 커밋 SHA 태그로 바꾸는 것만으로 "수동 restart"라는 단계가 사라진다.

## 7. 요약

1. 컨테이너는 **앱 + 실행환경**을 한 상자에 포장해 환경 차이 문제를 없앤다.
2. VM과 달리 **커널을 공유**하므로 가볍고 빠르다. 대신 **CPU 아키텍처를 맞춰야** 한다.
3. **이미지 : 컨테이너 = 템플릿 : 인스턴스** (Launch Template : EC2 인스턴스와 동일 구조)
4. 이미지는 **레이어**로 쌓이며 캐시·공유된다 → Dockerfile 명령 순서가 빌드 속도를 좌우한다.
5. 레지스트리는 **이미지판 Git 원격 저장소**. 비공개 레지스트리는 개발자와 **클러스터 양쪽 모두** 인증이 필요하다.
6. **`latest`는 덮어쓰기 가능한 이름표**이므로 GitOps와 상성이 나쁘다.
   → **커밋 SHA 기반 불변 태그**로 전환하는 것이 자동 배포의 전제 조건이다.
